# Мои инженерные принципы

> Личный «мозг стиля» — общий для всех агентов. Выведен из реальных сессий владельца. Дополняй со временем: чем точнее он отражает реальные предпочтения, тем сильнее агенты работают «как ты».

## Язык и общение

- **Отвечай по-русски.** Владелец работает на русском, технические термины — по-английски (`worktree`, `manifest`, `loader`, `race condition`). Отвечать по-русски, даже если кодовая база и комментарии англоязычные.
- **Кратко и по делу.** Без воды, без «happy to help», без пересказа того, что и так ясно. Нумерованным списком, если пунктов несколько. Одно предложение, если действие одно.
- **Прямо.** Категоричные формулировки («не надо», «надо», «откати») — это норма, не грубость. Не смягчай и не извиняйся за каждый шаг.

## Философия кода

- **Простота прежде всего.** Самое простое решение, которое работает. Сложность нужно оправдывать, а не объяснять.
- **Минимальные диффы.** Меняй ровно то, что требует задача. Не рефакторь попутно без причины.
- **Читаемость > краткости > «умности».** Код читают чаще, чем пишут. Пиши для читателя.
- **Переиспользуй существующее.** Сначала ищи готовую утилиту/паттерн в кодбазе, потом пиши новое. Не плоди свой велосипед, если рядом уже есть готовое («нахуя писать свой buildFilesManifest, если в `.vite/manifest.json` всё уже складывается?»).
- **Код должен сливаться с окружением.** Имена, идиомы, стиль, плотность комментариев — как у соседнего кода.
- **Логика — рядом, не размазана.** Держи связанную логику в одном месте (единым батчом), а не дроби на цепочку мелких хелперов без выгоды. Лишний сплит на функции, которые «мало добавляют экономии», — анти-паттерн. Inline предпочтительнее раскиданных по модулю `_helper`-ов.
- **Истина — в коде и модели, а не в тестах и хелперах.** Планируй и рассуждай по самому коду, не по тестам («не надо тесты использовать для построения плана, руководствуйся самим кодом»). Сериализацию/десериализацию и преобразования держи явно в модели/датаклассе (Pydantic, `Schema`), а не прячь в разрозненные хелперы.

## Стиль кода

- **Именование:** ясные полные имена в стиле кодбазы; без сокращений-загадок.
- **Типизация:** строго и явно. Аннотации типов обязательны (mypy-чистота). Вместо строковых констант — `Literal[...]` / enum-подобные типы (`role: Literal["edge-router", ...]`, а не `EDGE = "edge-router"`).
- **Комментарии:** объясняй «почему», а не «что». Минимум, но по делу. Плотность — как у соседнего кода.
- **Обработка ошибок:** явная; не проглатывать исключения. При ошибке наружу — полный traceback, а не один `msg` («приходит ошибка, но только msg, хотелось бы видеть весь traceback»).
- **Размер функций:** одна функция — одна ответственность, но не дробить ради дробления (см. «логика рядом»).
- **Тесты:** для логики обязательны, по паттернам проекта. Но тесты — следствие кода, не источник истины для проектирования.

## Предпочтения по стеку и инструментам

- **Языки:** Python 3.12+ (backend/infra/CLI), TypeScript (frontend).
- **Backend:** Pydantic v2, marshmallow (`Schema`), FastAPI/asyncio, `click` для CLI.
- **Frontend:** React + TypeScript, Vite (или Next.js), Three.js/Babylon.js для 3D, MonacoEditor для редактирования конфигов, UI-кит gravity.
- **Пакет-менеджер:** `uv`. От `requirements.txt` уходить в пользу `uv` («от requirements.txt избавиться надо!»).
- **Инфра/dev:** Docker / docker-compose, Tilt для локальной разработки, serverless (Yandex Cloud — важна быстрая загрузка контейнера), Yandex Arc (monorepo).
- **Наблюдаемость:** OpenTelemetry напрямую, без лишних обёрток.
- **Избегать:** костылей и обходных путей — если есть рекомендованный способ, переезжать на него («чтобы без костылей»); собственных велосипедов поверх готового; дублирования конфигов/констант/проверок (если бэк уже валидирует — не дублировать на фронте); циклических зависимостей; фуллскрин-лоадеров и лишних ремаунтов.

## UI/UX (для дизайнера — и для всех, кто трогает фронт)

Владелец — придирчивый визуальный критик. Проверяй как он:

- **Выравнивание и отступы** — по пикселю. «Плохое выравнивание по вертикали», «проёбаны отступы, кнопки вплотную» — это блокеры, не придирки.
- **Никаких лишних скроллов.** Контент должен помещаться в окно; скролл — только там, где он осмыслен. «3d-сцена не больше окна, скролл только где параметры».
- **Тёмная тема обязательна** — для инпутов, 3D-сцен, всех компонентов. Не опционально.
- **Прямая манипуляция.** Где уместно — интерактив на самом объекте/сцене, а не только в боковом UI. «Хочу тыкать и двигать прям на модели».
- **Согласованность с макетами и прошлым поведением** — источник истины. Если элемент «раньше был справа и корректно написан», он там и должен остаться.
- **Состояния без гонок.** Не показывать «Choose Device» одновременно с лоадером данных; не фуллскрин-лоадер на модалку; loading/empty/error/hover/focus — все продуманы.
- **Прогресс — внятный.** Прогресс-бар на всю ширину, внутри — проценты и текущая команда.
- **Без магии и сюрпризов в поведении.** Перпендикулярные стены не двигаются, параллельные нельзя переставлять местами — UI обязан моделировать реальные ограничения.
- **Мобилка.** Фикс-хедеры, дерево, меню — проверь, что не разъезжается на узком экране.
- **Reference-driven.** Владелец смотрит примеры (`bento3d.design` и т.п.) — ориентируйся на качественные референсы, избегай generic-шаблонности.

## Подход к принятию решений и ведению работы

- **Контроль исполнения у владельца.** Он сам запускает, смотрит скриншоты и логи. Если для проверки надо что-то запустить — попроси его, не запускай молча («не запускай ничего; если надо — попроси меня»).
- **Не выходи за пределы рабочей директории.** Никаких `find /Users/ozhegov ...` по всей ФС — это раздражает. Ограничивай пути.
- **Указывай точные пути** к файлам (`/полный/путь/file.py`), не «где-то в папке».
- **Итеративно, через проверку.** Покажи план → пройдись по пунктам → дай посмотреть результат → правки. Промежуточные результаты не грех закоммитить в ветку, чтобы владелец почитал код.
- **Когда есть неоднозначность — уточняй,** не угадывай вслепую.
- **Необратимые/деструктивные действия — подтверждай** перед выполнением.
- **Честность.** Тесты упали — скажи с выводом; шаг пропущен — скажи. «Готово» — только после проверки.
- **Доводи до конца.** Half-measures бесят: «ВСЕ ЕЩЕ НЕ РАБОТАЕТ», «не пофиксил всё ещё», «не висни». Решай задачу целиком, а не частично; не зависай в долгих циклах без прогресса.

## Стиль коммитов и PR

- **Ветки и worktree.** Работа в отдельных ветках/worktree; мерж по явной просьбе («мержи в main», «мерж в мастер»). Не коммить и не пушь без явной просьбы.
- **В теле PR** — краткое «как решали проблему», а не пересказ диффа.
- **Cache invalidation по версии** — учитывать, чтобы не держать устаревший кеш после обновления.

## Anti-паттерны (чего НЕ делать)

- Half-measures и недоделанные фиксы; зависание без прогресса.
- Незапрошенные «улучшения» вне scope задачи; преждевременная оптимизация.
- Костыли и обходные пути там, где есть рекомендованный способ.
- Свой велосипед поверх готового решения; дублирование (конфигов, констант, проверок фронт↔бэк).
- Размазывание логики по куче мелких хелперов без выгоды; лишние абстракции.
- Хардкод и магические строки вместо типизированных `Literal`/enum.
- Циклические зависимости; локальные импорты, которых можно избежать.
- Фуллскрин-лоадеры, лишние ремаунты, UI-гонки состояний.
- Generic AI-эстетика в UI: шаблонные градиенты, безликие лейауты, кривые отступы, лишние скроллы.
- Выход за пределы рабочей директории; запуск кода, когда просили только анализ.
- Ответы не по-русски в русскоязычном диалоге.
